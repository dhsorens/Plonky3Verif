# Aeneas extraction status (2026-05-28)

A re-evaluation of [Aeneas](https://github.com/AeneasVerif/aeneas) (+ Charon)
as an extractor for the Plonky3 crates in this repo. The previous attempt
was abandoned several months ago; aeneas has made substantial progress
since, so this is a fresh look.

## Tool versions

| Tool   | Version                                             | Source                              |
| ------ | --------------------------------------------------- | ----------------------------------- |
| aeneas | `42776dac` (latest `main` as of 2026-05-28)         | `~/devel/fv-tools/aeneas`           |
| charon | `0.1.198` at commit `103daa06` (aeneas's pinned charon) | `~/devel/fv-tools/aeneas/charon`    |

Both built from source via `gmake build` after `git pull` on aeneas (165
commits) and `git checkout` of charon to the aeneas-pinned commit.

## Invocation that worked

```
# in the crate dir, e.g. util/
charon cargo --preset aeneas --dest-file /tmp/.../p3_util.llbc
aeneas /tmp/.../p3_util.llbc -backend lean \
  -dest /tmp/.../lean -split-files -lean-default-lakefile
```

Notes:
- `--preset aeneas` is **required**. Without it, aeneas chokes on the LLBC
  with `Hash-consing key not found; there is a serialization mismatch
  between Rust and OCaml`. The aeneas preset turns on the deduplicated
  serialization format the OCaml side expects.
- `charon cargo` only emits a `.llbc` for the crate it actually re-compiles.
  If you run it again with cached artifacts, you get no new file. To force
  re-extraction of a downstream crate use `CARGO_TARGET_DIR=<scratch>` or
  `cargo clean -p <pkg>` first.

## Issue #1 — charon panics on `p3-field`

```
thread 'main' panicked at src/ast/types_utils.rs:1244:59:
called `Result::unwrap()` on an `Err` value: GenericsMismatch
```

This is the single biggest blocker. Until it's fixed, no crate transitively
depending on `p3-field` can be extracted from a clean target dir.

Worth filing upstream. Repro:

```
cd Plonky3Verif/field
CARGO_TARGET_DIR=/tmp/charon-field charon cargo --preset aeneas \
  --dest-file /tmp/charon-field/p3_field.llbc
```

## Issue #2 — trait associated constants are not looked up

Recurring error across `koala-bear` (1595 errors) and `poseidon2` (1170):

```
[Error] Could not find: trait_const_id:
  trait_decl: p3_field::field::PrimeCharacteristicRing (id: 20),
  const name: ZERO   (also ONE, TWO, NEG_ONE, GENERATOR, …)
[Error] Could not find: trait_parent_clause_id:
  trait_decl: p3_field::field::Field (id: …), clause_id: …
```

Anything that mentions `T::ZERO`/`T::ONE`/etc. on a `PrimeCharacteristicRing`
or `Field` bound fails to resolve. Combined with parent-clause lookups,
this is what produces the ~1500 cascading "Internal error: please file an
issue" entries in the koala-bear log.

This may interact with Issue #1 — perhaps charon is failing to record
these names because of the same `GenericsMismatch` it eventually panics on
in clean builds. Confirm by inspecting `core_models`-style stubs.

## Issue #3 — generic associated types (GATs)

Charon's own warnings on `koala-bear` include:

```
warning: Could not compute the value of (...::unpack_iter_ty::[…])::Item
on a GAT: lifting associated types can fail in the presence of GATs.
To fix this, `--exclude` this trait.
```

And on the aeneas side:

```
[Error] Can not extract trait associated types with parameters
```

These come from `field/src/op_assign_macros.rs` (`unpack_iter_ty`) and a
handful of related GAT-using traits. Aeneas has no GAT support yet.

## Issue #4 — raw pointers, inline asm, transmute

`util` hits:

```
[Error] Aeneas does not yet support dereferencing raw pointers.
[Error] Unsupported operation: &raw mut (*(v@N), ())
```

The `core::arch::asm!` blocks in `util/src/lib.rs:402` and
`util/src/transpose/rectangular.rs:87` are flagged as warnings by charon
("Inline assembly is not supported") — the produced LLBC simply elides
them, which is fine for verification scope but means the asm fast paths
are unverified by construction.

## Issue #5 — `FnOnce` / `core::option::Option::map` name clashes

`util`:

```
[Error] Name clash detected: …  "core.ops.function.FnOnce.mk"
[Error] Name clash detected: …  "core.option.Option.map"
[Error] Name clash detected: …  "core.option.Option.unwrap"
[Error] Name clash detected: …  "core.ops.function.FnOnce"
```

Aeneas emits the same Lean symbol for multiple distinct Rust monomorph-
isations of these standard-library items, which produces an unsound /
broken file. Likely an aeneas extractor bug around handling of `FnOnce` /
closure types.

## Issue #6 — mixed-recursive declaration groups

`koala-bear`:

```
[Error] Mixed declaration groups (which contain both type and function
declarations or functions and traits, for instance) are not supported
yet: [78, 111]
```

8 occurrences. These are dependency cycles spanning trait/type/function
boundaries that the current aeneas frontend can't lay out in Lean.

## Issue #7 — `Not_found` uncaught exception at end of run

Both `util` and `koala-bear` end with:

```
Uncaught exception:  Not_found
Raised at Stdlib__Map.Make.find in map.ml:141
Called from Aeneas__Translate.trait_impl_is_builtin in Translate.ml:1017
```

Aeneas still writes whatever partial files it has at this point, but the
exit code is nonzero. A defensive `Map.find_opt` in `trait_impl_is_builtin`
would let the run finish cleanly instead of throwing.

## Issue #8 — Closures / lambdas don't extract

`koala-bear` warnings:

```
[Warn ] Could not translate type decl
  'p3_field::field::RawDataSerializable::into_byte_stream::closure
   because of previous error
```

(also `into_parallel_byte_streams::closure`, `unpack_iter::closure`, …)

Each of these comes paired with `[Error] Internal error: please file an
issue`, suggesting aeneas hits an unhandled case while constructing the
closure environment for these traits. There are dozens of these.

## What the closest-to-working extractions look like

- **`keccak`** (1154 lines, 3 errors): the cleanest sample. Errors are:
  - `Could not find: type_var_id: …`
  - `Region ids should not be visited directly`
  Both look like internal invariant violations in aeneas rather than
  fundamental modeling gaps.

- **`util`** (1541 lines, 32 errors): produces sizeable real Lean but
  with raw-pointer / `FnOnce` gaps in the surface area. A meaningful
  subset (bit manipulation, basic indexing utilities) likely extracts
  correctly and could be verified standalone.

## Recommendation

Don't try to drive a Plonky3 verification campaign through aeneas yet.

What to do in order, easiest first:

1. **File the `field` panic upstream** (charon `GenericsMismatch` at
   `types_utils.rs:1244`). Until that's fixed, every downstream crate is
   compromised by the cached-vs-fresh build distinction, which is a
   nightmare for CI.
2. **Try `keccak` as a smoke test.** It's the closest-to-clean sample and
   has no Plonky3 dependencies; if it can be pushed over the finish line
   (`-exclude`/`-include` to dodge the two remaining errors), it's a
   reasonable demonstrator while the field issues are addressed.
3. **`util` is the next-best candidate** for a partial extraction —
   exclude the raw-pointer-using `transpose` module and the inline-asm
   spinloop, and aeneas might produce verifiable Lean for the rest.
4. The cryptographic core (`field`, `monty-31`, `koala-bear`, `poseidon*`,
   `challenger`) is currently **blocked on** (a) Issue #1, (b) trait
   associated constant resolution (Issue #2), and (c) GAT support
   (Issue #3). All three are aeneas/charon-side fixes; nothing useful
   we can do from the Plonky3 source side except possibly rewriting
   trait shapes, which would be a major upstream change.

In the meantime, the canonical hax-based workspace at
`koala-bear/proofs/lean/extraction/` remains the working verification
target.

## Reproducing the survey

Use the committed script at `aeneas-obstructions/run-extract.sh`:

```
# from the repo root:
aeneas-obstructions/run-extract.sh <crate-dirname>
```

Outputs land in `/tmp/aeneas-extract/<crate>/`:
- `charon.log` — full charon stdout/stderr
- `aeneas.log.clean` — ANSI-stripped aeneas output
- `lean/` — emitted `*.lean` files (partial on error)
- `<crate>.llbc` — the intermediate LLBC

Logs from this run are preserved under `/tmp/aeneas-extract/` (not in
the repo) so they can be re-read or grepped without re-running anything.

### What to file upstream

In priority order (highest blocker first):

1. **charon: `GenericsMismatch` panic at `src/ast/types_utils.rs:1244:59`
   when compiling `p3-field`.** The unwrap on a `Result<_, GenericsMismatch>`
   in that file is hitting a real case in Plonky3's trait shape. Likely
   a near-cousin of [#1159](https://github.com/AeneasVerif/charon/issues/1159)
   (also a panic inside trait-impl translation) but the failure site
   and error type differ.
2. **aeneas: `trait_const_id` lookup fails for trait-bound associated
   constants** (`T::ZERO`/`T::ONE`/`T::NEG_ONE`/`T::GENERATOR` on
   `PrimeCharacteristicRing` / `Field` bounds). [#794](https://github.com/AeneasVerif/aeneas/issues/794)
   covers a related-but-distinct extraction-correctness bug for default
   methods using associated constants; this one is a lookup failure
   that produces ~1500 cascading errors in `koala-bear`.
3. **aeneas: uncaught `Not_found` from `Map.find` in
   `Aeneas__Translate.trait_impl_is_builtin` at `Translate.ml:1017`.**
   Defensive `find_opt` would let the run finish cleanly with whatever
   partial files it has. Easy patch.

The remaining issues (#3, #4, #5, #6, #8) are all on the upstream radar —
no new tickets needed, just track existing ones.
