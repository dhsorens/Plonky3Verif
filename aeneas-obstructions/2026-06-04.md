# Aeneas extraction status (2026-06-04)

Third periodic survey (see [`../AENEAS.md`](../AENEAS.md)). **The #1 blocker
moved for the first time.** charon [#1263](https://github.com/AeneasVerif/charon/pull/1263)
(fixes [#1260](https://github.com/AeneasVerif/charon/issues/1260)) landed today
and **eliminates the `GenericsMismatch` panic** — so the five crates that
previously panicked transitively on `p3-field` (`monty-31`, `challenger`, `mds`,
`symmetric`, `merkle-tree`) now reach aeneas and extract partially. But the
blocker *moved* rather than cleared: `field`/`air`/`commit` now hit a **new
charon stack overflow** in the same associated-type-lifting pass (on
`p3-field`'s `PackedFieldExtension`/`BinomialExtensionField` impls), and two of
the newly-reachable crates expose a **new aeneas `Not_found`**. Net: real
forward progress (7 → 10 crates produce a `.llbc`), still not usable for a
campaign.

> **Toolchain note — non-standard charon build this run.** aeneas `main`
> (`5c70d9b0`) still pins charon `a535e914` (the pre-#1263 commit) in
> `charon-pin`, so a by-the-book build would *not* include today's fix. To
> survey the fix we built **charon at `origin/main` `985e04c6`** (which contains
> #1263) and rebuilt aeneas's OCaml against that charon's `charon-ml` so the LLBC
> serialization stays consistent. Smoke-tested `util` first: no
> "Hash-consing key not found" mismatch, so the override is sound. Once aeneas
> bumps its pin past #1263 this becomes the normal path.
